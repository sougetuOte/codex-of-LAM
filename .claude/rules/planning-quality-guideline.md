# Planning Quality Guideline — 仕様・設計・タスクの品質基準

PLANNING フェーズで作成する成果物（specs, design, tasks）の品質判断基準。
`/clarify` スキル、`spec-template`、`design-architect`、`task-decomposer` と併用する。

## 1. Requirements Smells（仕様の曖昧さ検出）

仕様書に以下の「危険な単語」が含まれていたら、計測可能な定義に置き換える。

### 危険な単語リスト

| カテゴリ | 例 | 対処 |
|:--------|:---|:-----|
| 曖昧な形容詞 | 「適切な」「十分な」「高速な」「柔軟な」「容易な」 | 数値・条件で定義（「200ms 以内」等） |
| 計測不能な性能 | 「効率的」「堅牢」「スケーラブル」 | 具体的メトリクスに置換（例: 「堅牢」→「可用性 99.9%」、「効率的」→「O(n log n) 以下」、「スケーラブル」→「10K 同時接続に対応」） |
| 逃げ道 | 「可能であれば」「必要に応じて」「等」「など」 | 条件を明示するか削除 |
| 無制限スコープ | 「全て」「常に」「決して〜ない」 | 本当に例外なしか検証 |
| 曖昧な指示語 | 「それ」「これ」「その」（指示先が不明） | 具体的な名詞に置換 |
| 複合要件 | 「A および/または B」 | 個別の要件に分割 |

### 仕様書レビュー時のフロー

```
危険な単語を検出 → ユーザーに具体化を提案 → 置換後に再確認
```

## 2. RFC 2119 キーワード（義務レベルの統一）

仕様書の要件文には、以下のキーワードで義務レベルを明示する。

| キーワード | 意味 | 用途 |
|:----------|:-----|:-----|
| MUST / MUST NOT | 絶対的な要件・禁止 | 違反＝バグ |
| SHOULD / SHOULD NOT | 推奨・非推奨（正当な理由があれば逸脱可） | 違反＝要検討 |
| MAY | 任意（実装者の裁量） | 違反にならない |

### 適用ルール

- 1つの要件文に義務レベルの混在を避ける（「MUST かつ SHOULD」は分割）
- 義務レベルのないふわっとした要件文（「〜できるようにする」）は、いずれかに分類してから記述
- BUILDING フェーズでの優先順位: MUST → SHOULD → MAY

## 3. Design Doc チェックリスト（設計書の必須セクション）

設計書に以下のセクションが欠けていたら、起草段階で補完する。

| セクション | 目的 | 欠落時のリスク |
|:----------|:-----|:-------------|
| Problem Statement | 何を解決するか | 解法先行（技術ありきの設計） |
| Non-Goals（非スコープ） | 何をやらないか | スコープクリープ |
| Alternatives Considered | 他にどんな選択肢があったか | 決定の根拠不明 |
| Success Criteria | 何をもって成功とするか | 完了判定不能 |

### アンチパターン

- 「Why」より「What」が多い設計書 → 動機を補強
- 代替案が空の設計書 → 最低1つの却下案を記述
- 成功基準が計測不能 → 数値化または観測可能な条件に

## 4. SPIDR タスク分割パターン

タスクが大きすぎる場合（目安: 1セッションで完了しない）、以下の5軸で分割を検討する。

| 軸 | 分割方法 | 例 |
|:---|:--------|:---|
| **S**pike | 調査タスクを先行させ、結果に基づき分割 | 「ライブラリ A vs B の比較」→「選定結果に基づく実装」 |
| **P**aths | ユーザー経路・ワークフローで分割 | 正常系 / 異常系 / エッジケース |
| **I**nterfaces | インターフェース種別で分割 | API / CLI / UI |
| **D**ata | データ操作で分割 | Create / Read / Update / Delete |
| **R**ules | ビジネスルールで分割 | バリデーション A / バリデーション B |

**注意**: 水平分割（「DB層だけ」「UI層だけ」）は避け、**垂直分割**（全層を薄く貫通）を優先する。

## 5. WBS 100% Rule（仕様⇔タスクのトレーサビリティ）

タスク定義完了時に以下を検証する:

```
仕様の全 FR/NFR に対応するタスクが存在するか？ → 漏れなし
全タスクが仕様の FR/NFR にトレースできるか？   → 孤児なし
```

| 状態 | 意味 | 対処 |
|:-----|:-----|:-----|
| 仕様にあるがタスクにない | 実装漏れ（Gap） | タスクを追加 |
| タスクにあるが仕様にない | 孤児作業（Orphan） | 仕様を追加するか、タスクを削除 |

## 6. Example Mapping（`/clarify` での活用）

要件の明確化時に、以下の4層構造で整理する:

```
要件（Yellow）
  └─ ルール（Blue）: この要件を満たす条件
       └─ 具体例（Green）: ルールを検証する具体的シナリオ
       └─ 具体例（Green）: 別のシナリオ
  └─ ルール（Blue）: 別の条件
       └─ 具体例（Green）
  └─ 未解決質問（Red）: まだ決まっていないこと
```

### 判断基準

- **Red（未解決質問）が1件以上残っている要件は、解決してから実装に進む**
- Green（具体例）が1つもないルールは、テスト不能の兆候
- 具体例が BDD の Given-When-Then に変換できれば、受入条件として採用

## 根拠

- IEEE 830 SRS 品質属性（correct, unambiguous, complete, consistent, verifiable）
- RFC 2119 / RFC 8174（IETF 要件キーワード）
- Google / Stripe Design Doc practices
- SPIDR Story Splitting（Mountain Goat Software）
- WBS 100% Rule（PMI）
- Example Mapping（Matt Wynne, Cucumber Ltd.）

# Clarify — 文書精緻化インタビュースキル

## 目的

技術文書（spec、design、ADR、tasks）の曖昧さ・矛盾・欠落を
構造化されたインタビューで特定し、文書品質を引き上げる。

## 使い方

```
/clarify <ファイルパス>           # 指定文書を精緻化
/clarify docs/specs/foo-spec.md  # spec を精緻化
/clarify docs/design/foo.md      # design を精緻化
```

引数なしの場合は、対象ファイルをユーザーに確認する。

## 実行フロー

### Phase 1: 文書分析

対象文書を精読し、以下の観点で問題を抽出する:

#### 1.1 曖昧さ検出

以下のパターンを検出する:

| パターン | 例 |
|---------|-----|
| 曖昧な修飾語 | 「適切に」「十分な」「必要に応じて」「等」 |
| 未定義の用語 | 初出の専門用語に定義がない |
| 主語の欠落 | 「処理される」（誰が？） |
| 数値の欠落 | 「大量の」「高速な」（具体的な数値は？） |
| 条件の欠落 | 「エラー時は対処する」（どう対処？） |
| スコープの曖昧さ | 「関連するファイル」（どの範囲？） |

#### 1.2 矛盾検出

- 同一文書内で異なる記述がある箇所
- 関連文書（spec ↔ design ↔ tasks）間の不一致
- 用語の不統一（同じ概念に異なる名前）

#### 1.3 欠落検出

| 文書種別 | チェック項目 |
|---------|------------|
| **spec** | 機能要件に対応する非機能要件があるか、エラーケースの定義、境界値の定義 |
| **design** | 全 FR が設計に反映されているか、データモデル定義、エラーハンドリング方針 |
| **ADR** | 却下した選択肢とその理由、影響範囲、リスク |
| **tasks** | 全設計コンポーネントのタスクカバレッジ、依存関係の完全性、受け入れ条件 |

#### 1.4 関連文書チェック（オプション）

対象文書が参照する他の文書（spec → design → tasks）を特定し、
文書間の整合性も分析対象に含める。

### Phase 2: 質問生成 + インタビュー

#### 2.1 質問の構造

抽出した問題を重要度でソートし、**1回あたり 3-5 問**に絞って提示する。

```
=== Clarify セッション: [ファイル名] ===
分析結果: 曖昧 X件 / 矛盾 Y件 / 欠落 Z件

--- Round 1 (5問) ---

Q1 [曖昧] Section 3.2
「適切なチャンクサイズ」とありますが、具体的な数値または範囲を
定義できますか？
  現状: 「適切なサイズに分割する」
  提案: 「2,000-4,000 トークンに分割する」

Q2 [欠落] Section 4
エラーケースが未定義です。静的解析ツールが未インストールの場合、
パイプラインはどう動作すべきですか？
  A) スキップして続行（WARN ログのみ）
  B) エラーで停止
  C) ユーザーに確認を求める

Q3 [矛盾] Section 2.1 ↔ Section 5
...

回答方法:
- 番号 + 回答（例: "1. 2000-4000トークン"）
- "skip" で後回し
- "任せる" でAI提案を採用
```

#### 2.2 回答パターンの処理

| ユーザー回答 | 処理 |
|------------|------|
| 具体的な回答 | 文書に直接反映 |
| 「わからない」「考えてない」 | Three Agents Model で選択肢を提示（下記参照） |
| 「skip」「後で」 | `[TBD]` マークを文書に記録し、次の質問へ |
| 「任せる」「おまかせ」 | AI が最適案を仮決定し `[仮決定: 理由]` マークで記録 |
| 「これは不要」 | 該当セクション/要件の削除を提案 |

#### 2.3 Three Agents 提案モード

ユーザーが「わからない」と回答した場合に自動発動:

```
Q2 について Three Agents 分析:

[Affirmative] A) スキップして続行
  → 柔軟性が高い。ツール未導入の新規ユーザーでも動作する

[Critical] B) エラーで停止
  → 安全側。未スキャンのコードがすり抜けるリスクをゼロにできる

[Mediator] 推奨: A)
  → 理由: LAM の既存 G5 チェックも pip-audit 未インストール時に
     PASS(skip) としており、一貫性がある

A/B/C のいずれかを選択、または別案を提示してください:
```

### Phase 3: 文書更新

ユーザーの回答を反映して文書を更新する。

- 回答済み: 曖昧な記述を具体的な記述に置換
- TBD: `<!-- [TBD] 質問内容 -->` としてコメントで残す
- 仮決定: `<!-- [仮決定] 採用案: 理由 -->` としてコメントで残す

### Phase 4: 完了判定

全質問の処理後:

```
=== Clarify 完了 ===
解決: X件 / TBD: Y件 / 仮決定: Z件

TBD 一覧:
- Section 3.2: チャンクサイズの上限
- Section 5: エラーリカバリ戦略

次のアクション:
- TBD が残っている場合: 再度 /clarify を実行
- 全解決の場合: 文書は精緻化完了
```

## 複数文書の横断チェック

`/clarify` に複数ファイルまたはディレクトリを指定した場合、
文書間の整合性チェックも実施する:

```
/clarify docs/specs/foo-spec.md docs/design/foo-design.md docs/tasks/foo-tasks.md
```

横断チェック観点:
- spec の FR が全て design に反映されているか
- design の全コンポーネントが tasks でカバーされているか
- 用語の統一（3文書間で同じ概念に同じ名前が使われているか）
- 参照パスの正確性

## 設計原則

1. **1回の質問数は 3-5 問に制限** — ユーザーの認知負荷を下げる
2. **質問には必ず提案を添える** — 空白から考えさせない
3. **止まらない設計** — ユーザーが答えられなくても TBD/仮決定で進む
4. **重要度順** — Critical な曖昧さから先に解消する
5. **文書を直接更新** — 回答は口頭で終わらせず、必ず文書に永続化する

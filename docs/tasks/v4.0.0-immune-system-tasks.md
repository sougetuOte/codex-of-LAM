## 概論

v4.0.0 の実装は4つの Wave に分かれており、各 Wave は独立してロールバック可能である。各タスク（TASK）は 1PR で完結する粒度に分解されている。

**全タスク数**: 22 タスク
**全 PR 数**: 22 PR
**依存関係**: Wave 0 → Wave 1（並列実行不可）→ Wave 2, 3（順序柔軟）→ Wave 4

## 実装時の注意事項

### 共通事項

1. **ファイル既読**: BUILDING フェーズ開始時に、変更対象の既存ファイルを全て読み込むこと
2. **仕様突合**: 実装内容と仕様書（要件定義書 + 設計書）の文字単位での一致を確認（TDD 品質チェック R-1）
3. **ドキュメント同期**: 実装と仕様書の乖離を検出したら即座に仕様書を更新（S-4 ルール）
4. **テストカバレッジ**: hook スクリプト（.sh）は機能テストで検証。ドキュメントはテンプレートフォーム確認

### Wave 別注意事項

**Wave 0**: KPI と ループログスキーマは、後続の Wave で参照されるため、仕様の曖昧さを排除すること

**Wave 1**: PreToolUse hook は LAM の核となるため、ファイルパス判定ロジックの正確性が重要。`/auditing` フェーズでの PG 修正許可は v3.x 大幅変更のため、phase-rules.md が明確に反映していることを確認

**Wave 2**: Stop hook と PostToolUse hook は並行実装されるため、状態ファイル（.claude/lam-loop-state.json）のスキーマが両者で一貫していることを確認。full-review の Phase 1-5 フローは既存との互換性が重要

**Wave 3**: doc-writer エージェントへのタスク割り当ては Wave 1 の権限機構に依存。Wave 2 の PostToolUse hook によって doc-sync-flag が正確に設定されていることが前提

**Wave 4**: TDD 内省は、Wave 1 の PM級承認フロー（TASK 1-1 の権限等級）と Wave 2 の PostToolUse hook 両方に依存。信頼度モデルの閾値（初期値 3回）は運用データに基づいて調整する余地あり

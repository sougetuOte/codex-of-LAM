AI 開発の「あるある」

  • 「仕様を確認せず勝手に実装を始める」
  • 「ドキュメントと実装がどんどん乖離する」
  • 「品質基準が曖昧で、何をもって完了とするか不明」
  • 「セッションが切れるとコンテキストが消失する」

LAM のアイデンティティ

Living Architect（生きた設計者）
プロジェクト全体の整合性を維持する

Gatekeeper（門番）
仕様なし実装、テストなし実装を許さない

責務は「コードを書くこと」よりも
「プロジェクトの健全性を守ること」

PLANNING

許可: 設計ドキュメント出力

禁止: コード生成

BUILDING

必須: 仕様確認、TDD 厳守

禁止: 仕様なし実装

AUDITING

許可: PG/SE 級の修正

禁止: PM 級（仕様・設計変更）の直接修正

コア原則

Active Retrieval

記憶に頼らず、常にファイルを確認する。「覚えているはず」を仮定しない。

Zero-Regression

変更前に影響範囲をシミュレーション。最も遠いモジュールへの波及を事前検証。

Living Documentation

実装とドキュメントは同一の不可分な単位。コードだけ変更は許されない。

Hierarchy of Truth

User Intent > Architecture > Specs > Code。判断に迷ったらこの順で。

Quick Start

1
GitHub で Use this template または git clone
Codex App と Git が基本（詳細は QUICKSTART.md）
2
PLANNING phase で要求仕様を相談・生成
3
要件確定後、AGENTS.md / README 等をプロジェクトに適応（.codex/ と docs/internal/ は基盤として維持）
4
design / ADR / tasks を生成 → BUILDING phase で TDD 実装
5
AUDITING phase で review pane と検証結果を確認 → Green State 達成

権限等級 — 判断の自動化

PG級: 自動修正、報告なし

AI が import を整理 → 自動修正、報告なし

「人間が関与する必要のない修正」

SE級: 修正後に報告

AI がエラーハンドリングを改善 → 修正後に報告

「技術的判断はあるが、仕様には影響しない」

PM級: ユーザーに承認を求める

API のレスポンス形式を変更したい → ユーザーに承認を求める

「仕様に影響する変更は、人間が決める」

午後 — /full-review

You

/full-review src/auth/

Living Architect

Stage 0: 初期化 + Scale Detection... Stage 2: 並列監査開始...

iter 1: 5件発見 → 5件修正 → 再スキャン
iter 2: 1件発見 → 1件修正 → 再スキャン
iter 3: 0件発見 → ✅ Green State

人間は他の作業をしていてOK。AI が自動で収束させる

Green State 5条件

  • G1: テスト全パス
  • G2: lint エラーゼロ
  • G3: 対応可能 Issue ゼロ
  • G4: 仕様差分ゼロ
  • G5: セキュリティチェック通過

重要: Issue を修正した後ではなく、スキャンで 0件 が Green State

コンテキスト管理

問題: 長時間作業するとコンテキストウィンドウが逼迫する

1. StatusLine でコンテキスト残量を常時監視

2. 残量 20% 以下で AI が自動的に /quick-save を推奨

3. /quick-save → SESSION_STATE.md に軽量保存（3-4% 消費）

4. 新しいセッションで /quick-load → 即座に復帰

一日のまとめ

素の AI LAM
復帰 毎回ゼロから説明 /quick-load で即復帰
中断 コンテキスト消失 /quick-save & /quick-load で即復帰
品質 人間がチェック Green State で自動判定
コミット メッセージを考える /ship で自動分類
保存 切れたら全ロスト /quick-save で状態保存

人間は創造的な意思決定に集中。定型作業は AI に任せる

リリース

LAM テンプレートには汎用的な /release スキルが含まれている。
CHANGELOG 更新 → git tag → GitHub Release を自動化する。

Living Architect

リリースフローはプロジェクトごとに異なります。CI/CD パイプライン、デプロイ先、承認フロー等に合わせて 自社用の release スキルを作成してください。

スキルは .claude/skills/ に追加するだけ。LAM が自動的に認識する

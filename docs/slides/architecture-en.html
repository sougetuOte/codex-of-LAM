Monitoring all tool operations to protect the project with 3 layers of defense

PreToolUse — The Gatekeeper

File path-based permission checks

  • Changes to docs/specs/, docs/adr/, .claude/rules/ → PM level (approval required)
  • Changes to src/ → SE level (report after fix)
  • Formatting fixes, etc. → PG level (auto-approved)

Every write operation is permission-checked before execution

PostToolUse — The Learner

  • Auto-detection of test failure → success patterns
  • Recording to tdd-patterns.log (PG level, automatic)
  • doc-sync-flag generation (on src/ changes)

Accumulating failure and success patterns to drive framework evolution

Stop Hook — Loop Control

  • State management via lam-loop-state.json
  • Green State evaluation (tests + lint)
  • Iteration limit (default: 5 rounds)
  • Full scan control (diff → full, two-stage)

A safety mechanism to prevent infinite loops and ensure convergence

Permission Levels PG / SE / PM

LevelMeaningActionExamples
PG Trivial fix Auto-fix allowed typo, import sorting, lint fix
SE Technical judgment Report after fix Add tests, internal refactor
PM Spec impact Approval required API change, rule change, spec change

When in doubt between PG and SE, round up to SE (err on the safe side)

Green State: 5 Conditions

#ConditionDetails
G1All tests passpytest / npm test, etc.
G2Zero lint errorsruff / eslint, etc.
G3Zero actionable issuesPG/SE level issues resolved
G4Zero spec driftdocs/specs/ aligned with implementation
G5Security passedDependency vulnerabilities + secret scan

Green State = scan finds 0 issues. Not "0 issues after fixes"

AoT (Atom of Thought)

Trigger conditions (any one applies)

  • Decision points: 2 or more
  • Affected layers: 3 or more
  • Viable options: 3 or more

Decompose complex decisions into independent Atoms,
MAGI deliberation per Atom → Reflection validates conclusions

Default operation relies on gates, planning, verification, and review. Codex hooks are a feature-flagged optional advanced path.

Approval gate — The Gatekeeper

Human review points are placed according to phase and change type.

  • requirements / ADR / design / tasks → proceed after approval
  • Implementation changes → TDD and focused verification
  • Publication, deletion, or distribution decisions → explicit release gate

Do not rely on automatic blocking alone; judge from conversation, plans, diffs, and verification evidence.

Verification evidence — The Learner

  • Record Red / Green / Refactor and verification commands
  • Move long history to docs/artifacts/ or docs/daily/
  • Keep only short resume facts in SESSION_STATE.md

Keep results visible so the next gate can judge from evidence.

Green State — Convergence

  • Required tests, lint, and static scans pass
  • Known blockers and residual risks are visible
  • Open decisions are not hidden in the next gate
  • Long work can be paused safely with quick-save

Code alone is not done. Verification evidence is part of completion.

Codex hooks — optional path

  • Enable lifecycle hooks with features.codex_hooks = true
  • Candidate events include PreToolUse, PermissionRequest, PostToolUse, UserPromptSubmit, and Stop
  • Stop may support a future Codex-native full-review pilot
  • Because hooks are not a complete enforcement boundary, do not make them the default public-template runtime yet

Today's substitute is context-harvest + magi + review pane + focused verification

Permission Levels PG / SE / PM

LevelMeaningActionExamples
PG Trivial fix Auto-fix allowed typo, import sorting, lint fix
SE Technical judgment Report after fix Add tests, internal refactor
PM Spec impact Approval required API change, rule change, spec change

When in doubt between PG and SE, round up to SE (err on the safe side)

Green State: 5 Conditions

#ConditionDetails
G1All tests passpytest / npm test, etc.
G2Zero lint errorsruff / eslint, etc.
G3Zero actionable issuesPG/SE level issues resolved
G4Zero spec driftdocs/specs/ aligned with implementation
G5Security passedDependency vulnerabilities + secret scan

Green State = scan finds 0 issues. Not "0 issues after fixes"

MAGI: Three independent minds deliberating as one (inspired by Evangelion)

AoT (Atom of Thought)

Trigger conditions (any one applies)

  • Decision points: 2 or more
  • Affected layers: 3 or more
  • Viable options: 3 or more

Decompose complex decisions into independent Atoms,
MAGI deliberation per Atom → Reflection validates conclusions

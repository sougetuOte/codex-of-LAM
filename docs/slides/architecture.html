通常運用は承認 gate、計画、検証、レビュー。Codex hooks は feature flag 配下の optional advanced path として扱う

承認 gate — 門番

phase と変更種別に応じて、人間の確認点を置く

  • requirements / ADR / design / tasks → 承認後に次へ進む
  • 実装変更 → TDD と focused verification で確認
  • 公開・削除・配布判断 → release gate として明示

自動ブロックだけに頼らず、会話・plan・差分・検証結果を見て判断する

検証記録 — 学習者

  • Red / Green / Refactor と検証コマンドを残す
  • 長い履歴は docs/artifacts/ や docs/daily/ へ逃がす
  • SESSION_STATE.md には次回復帰に必要な短い要点だけ残す

結果を見える形に残し、次の判断で読み返せるようにする

Green State — 収束判断

  • 必要なテスト・lint・静的 scan が通っている
  • 既知の blocker と残リスクが明示されている
  • 未解決の判断を次の gate に隠して持ち越さない
  • 作業が長くなる前に quick-save で安全に区切れる

コードがあるだけでは完了ではない。検証結果までが完了条件

Codex hooks — optional path

  • features.codex_hooks = true で lifecycle hooks を有効化できる
  • PreToolUse, PermissionRequest, PostToolUse, UserPromptSubmit, Stop が候補
  • Stop は将来の Codex-native full-review pilot に使える可能性がある
  • ただし complete enforcement boundary ではないため、公開 template の標準 runtime にはまだ入れない

現時点の代替は context-harvest + magi + review pane + focused verification

権限等級 PG / SE / PM

等級意味アクション例
PG 自明な修正 自動修正可 typo, import整理, lint修正
SE 技術判断あり 修正後報告 テスト追加, 内部リファクタ
PM 仕様影響あり 承認必須 API変更, ルール変更, 仕様変更

PG で迷ったら SE級に丸める（安全側に倒す）

Green State 5条件

#条件内容
G1テスト全パスpytest / npm test 等
G2lint エラーゼロruff / eslint 等
G3対応可能 Issue ゼロPG/SE級は修正済み
G4仕様差分ゼロdocs/specs/ と実装の整合性
G5セキュリティ通過依存脆弱性 + シークレットスキャン

Green State = スキャンで Issue が 0件。修正後に 0件ではない

MELCHIOR

科学者（推進者）

  • Risk

MAGI: 3つの独立した思考体による合議（エヴァンゲリオン由来）

AoT (Atom of Thought)

適用条件（いずれか該当）

  • 判断ポイントが 2つ以上
  • 影響レイヤーが 3つ以上
  • 選択肢が 3つ以上

構造化された協働フレームワークだ。

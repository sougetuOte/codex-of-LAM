## 1. 設計概要

### 1.1 レイヤーモデル

```
外殻層:  CLAUDE.md（憲法的プロンプティング、Hierarchy of Truth）
制御層:  .claude/rules/ + commands/ + agents/ + skills/（LAM 固有ロジック）
補完層:  hooks（PreToolUse / PostToolUse / Stop / PreCompact）
基盤層:  settings.json（allow / ask / deny）
```

hooks は Claude Code の実行パイプラインに介入する。settings.json は最も粗い粒度のガードレール。

### 1.2 データフロー概念図

```
ユーザー入力
    │
    ▼
[PreToolUse hook] ──→ PG: allow / SE: allow+log / PM: ask（承認ダイアログ）
    │
    ▼
Claude がツール実行
    │
    ▼
[PostToolUse hook] ──→ パターン記録(P3) / ドキュメント更新判定(P4) / ループログ(P2)
    │
    ▼
Claude が応答完了
    │
    ▼
[Stop hook] ──→ 収束条件判定: Green State? → block(継続) or allow(停止)
```

## 2. モデルルーティング設計

**ADR-0001** に基づく。

### 2.1 hooks でのモデル使用

| 判定レイヤー | handler type | model | timeout | 用途 |
|-------------|-------------|-------|---------|------|
| パスベース粗分類 | `command` | なし | 600s(default) | ファイルパス・ツール名で PG/SE/PM を判定 |
| 内容ベース判定 | `prompt` | `haiku` | 30s(default) | グレーゾーンの LLM 判定 |
| 深い検証 | `agent` | `sonnet` | 60s(default) | ファイル内容を読んで判定（tool access あり） |

**Opus は hooks で使用しない**（メインセッション専用）。

### 2.2 subagents でのモデル使用

| エージェント | model | 理由 |
|-------------|-------|------|
| test-runner | `haiku` | テスト実行・結果解析は軽量タスク |

既存スキルは `disable-model-invocation: true` か未指定のまま維持。v4.0.0 で新規追加するスキルのみモデルを指定する。

## 3. Hook 設計

### 3.1 hooks 設定の全体構成

`.claude/settings.json` の `hooks` セクションに以下を追加:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "hooks": [{ "type": "command", "command": "python3 \"$CLAUDE_PROJECT_DIR\"/.claude/hooks/pre-tool-use.py" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write|Bash",
        "hooks": [{ "type": "command", "command": "python3 \"$CLAUDE_PROJECT_DIR\"/.claude/hooks/post-tool-use.py" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": "python3 \"$CLAUDE_PROJECT_DIR\"/.claude/hooks/lam-stop-hook.py" }]
      }
    ],
    "PreCompact": [
      {
        "hooks": [{ "type": "command", "command": "python3 \"$CLAUDE_PROJECT_DIR\"/.claude/hooks/pre-compact.py" }]
      }
    ]
  }
}
```

> **書式（2026-03 公式ドキュメント準拠）**: 各イベント配列の要素は `{ matcher?, hooks: [{ type, command, ... }] }` 構造。`matcher` は正規表現文字列で、省略すると全ツールにマッチする。`Stop` / `PreCompact` は matcher 非対応（常に発火）。パスは `$CLAUDE_PROJECT_DIR` 変数を使用し、作業ディレクトリ変更に対応する。
>
> Claude Code は `command` / `http` / `prompt` / `agent` の4種の hook type をサポートする。MVP では `command` 型のみ使用。LLM 判定が必要な場合は hook の応答で `ask` に劣化させ、メインセッションの Claude に判断を委ねる。

### 3.2 PreToolUse hook（権限判定）

**目的**: ツール実行前に変更の PG/SE/PM 等級を判定し、適切なアクションを取る。

**入力** (stdin JSON):
```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "...",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Edit",
  "tool_input": { "file_path": "src/main.py", "old_string": "...", "new_string": "..." }
}
```

**判定ロジック** (`.claude/hooks/pre-tool-use.py`):

```
1. ツール名チェック:
   - Read/Glob/Grep → exit 0（読取りは常に許可）

2. ファイルパスベース判定:
   - docs/specs/*.md, docs/adr/*.md → PM級
   - .claude/rules/*.md → PM級
   - .claude/settings*.json → PM級
   - docs/ 配下（上記以外） → SE級
   - その他 → SE級（デフォルト、安全側）

3. AUDITING フェーズの特別処理:
   - .claude/current-phase.md を読み取り
   - AUDITING なら PG級のツール（lint修正等）は allow

4. 応答:
   - PG級 → exit 0（許可）
   - SE級 → exit 0 + ログ記録
   - PM級 → hookSpecificOutput 形式で ask（ネイティブ承認ダイアログ表示）:
     {"hookSpecificOutput": {"permissionDecision": "ask", "permissionDecisionReason": "PM級変更です。承認してください: [ファイルパス]"}}
```

**等級判定ログ** (P1-FR-5): 全判定結果を `.claude/logs/permission.log` に追記する。

```
2026-03-08T10:00:00Z  PG  Edit  src/main.py  "allow (path-based)"
2026-03-08T10:00:05Z  PM  Edit  docs/specs/api.md  "specs/ path"
```

MVP ではテキスト形式。完全実装で JSON 構造化。

**MVP での制約**: 純粋なパスベース判定のみ（command 型）。変更内容の意味理解は行わない。Wave 1 完了後の誤判定率計測でチューニングし、必要に応じて第2層（prompt + haiku）を追加。

**settings.json の allow と hook の優先順位**: Claude Code は `permissions` の評価（deny → ask → allow）を **hook より先に** 処理する。settings.json の allow に含まれるコマンドは PreToolUse hook に到達しない。したがって PG 級コマンド（`Bash(ruff format *)` 等）は settings.json で allow し、hook はそれ以外の操作を判定する。permissions ルールの書式はスペース区切り（例: `Bash(git log *)`）。

### 3.3 PostToolUse hook（パターン記録・ドキュメント更新判定）

**目的**: ツール実行後にパターン記録（P3）とドキュメント更新判定（P4）を行う。

**入力** (stdin JSON):
```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "...",
  "permission_mode": "default",
  "hook_event_name": "PostToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "pytest" },
  "tool_response": { "stdout": "...", "stderr": "...", "exitCode": 1 },
  "tool_use_id": "toolu_01ABC123..."
}
```

> **注意**: レスポンスフィールド名は `tool_output` ではなく `tool_response`。テスト失敗の検出には `tool_response` の内容をパースする。

**判定ロジック** (`.claude/hooks/post-tool-use.py`):

```
1. テスト結果の記録 (P3):
   - tool_name == "Bash" && command matches "pytest|npm test|go test"
   - exit_code != 0 → 失敗パターンを .claude/tdd-patterns.log に追記
   - exit_code == 0 && 前回失敗 → 「失敗→成功」パターンを記録

2. ドキュメント更新判定 (P4):
   - tool_name == "Edit" or "Write"
   - file_path が src/ 配下 → "doc-sync-needed" フラグをセット
   - フラグは /ship 実行時に参照

3. ループログ記録 (P2):
   - ループ状態ファイルが存在する場合、ツール実行結果をログに追記
```

**MVP での制約**: P3 のパターン記録はテキストベースのログ追記のみ。パターン照合は手動（Wave 4 で自動化）。P4 のドキュメント更新は `/ship` 時のみトリガー。

> **PostToolUseFailure イベント**: Claude Code はツール失敗時に `PostToolUse` とは別の `PostToolUseFailure` イベントを発火する。入力には `error`（文字列）と `is_interrupt`（boolean）が含まれる。MVP では未使用だが、TDD パターン検出の精度向上に活用可能（Wave 4 検討）。

### 3.4 Stop hook（自律ループ）

**ADR-0002** に基づき、独自実装（Ralph Wiggum の設計パターンを参考）。

**状態ファイル**: `.claude/lam-loop-state.json`

```json
{
  "active": true,
  "command": "full-review",
  "target": "src/",
  "iteration": 2,
  "max_iterations": 5,
  "started_at": "2026-03-08T10:00:00Z",
  "log": [
    {"iteration": 1, "issues_found": 5, "issues_fixed": 3, "pg": 2, "se": 1, "pm": 0, "test_count": 42},
    {"iteration": 2, "issues_found": 2, "issues_fixed": 2, "pg": 2, "se": 0, "pm": 0, "test_count": 42}
  ],
  "fullscan_pending": false,
  "tool_events": []
}
```

**入力** (stdin JSON):
```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "...",
  "permission_mode": "default",
  "hook_event_name": "Stop",
  "stop_hook_active": false,
  "last_assistant_message": "..."
}
```

> **`stop_hook_active`**: Stop hook の結果で Claude が継続中に `true` になる。このフラグを確認しないと**無限ループ**が発生するため、判定ロジックの最初でチェックする。

**判定ロジック** (`.claude/hooks/lam-stop-hook.py`):

```
0. 再帰防止チェック（最優先）:
   - stop_hook_active == true → 無条件 exit 0（状態ファイルの有無を問わず通常停止を許可。より安全側に倒した実装）

1. 状態ファイル確認:
   - .claude/lam-loop-state.json が存在しなければ exit 0（通過）

2. 反復上限チェック:
   - iteration >= max_iterations → 状態ファイル削除 + exit 0

3. コンテキスト残量チェック (P5-FR-3):
   - PreCompact が直近で発火していればコンテキスト圧迫と判断
   - → 状態ファイル削除 + exit 0（ループ停止、人間に委ねる）

4. Green State 判定:
   - テスト実行（pytest / npm test 等）→ 全パス?
   - lint 実行（設定があれば）→ エラーゼロ?
   - 全条件パス → フルスキャン実行（下記 3.4.1）→ 状態ファイル削除 + exit 0

5. エスカレーション条件チェック:
   - 前サイクルと同一 Issue が再発 → exit 0（人間に委ねる）
   - テスト数が減少 → exit 0（人間に委ねる）
   - PM級変更は PreToolUse hook (3.2節) で block 済みのため、ここでは重複チェックしない

6. 継続:
   - iteration をインクリメント
   - {"decision": "block", "reason": "Green State 未達。サイクル N+1 を開始。残Issue: X件"}
   - ※ Stop hook では decision:"block" が Claude の停止を阻止する（= 継続させる）
```

**ループ起動方法**: `/full-review` コマンド実行時に状態ファイルを生成。Stop hook はファイルの存在で自動的にループモードに入る。

#### 3.4.1 フルスキャン（最終サイクル）

Green State 達成後、最終確認としてフルスキャン（Phase 1 相当の全体監査）を実行する:

```
Green State 達成
  → フルスキャン（全対象ファイルを再監査）
  → 新規 Issue なし → 完了（exit 0）
  → 新規 Issue あり → 追加サイクルとして継続（block）
```

これにより AC-2.5（最終サイクルでフルスキャンが実行される）を満たす。

**再帰リスクについて**: Stop hook は Claude が停止するたびに発火する。`decision: "block"` で継続させた場合、次の停止時にも発火する。`stop_hook_active` フラグ（Claude Code が自動設定）と状態ファイルの存在チェックにより、意図しない無限ループを防止する。

### 3.5 PreCompact hook（強制セーブ）

**目的**: コンテキスト圧縮前に SESSION_STATE.md を自動保存。

**判定ロジック** (`.claude/hooks/pre-compact.py`):

```
1. SESSION_STATE.md に現在のタスク状態を書き出し
2. ループ中であれば lam-loop-state.json のログも保存
3. exit 0（常に圧縮を許可）
```

**制約**: command 型のみ対応。hook handler に `"async": true` を設定することで非同期実行が可能（圧縮をブロックしない）。ただし MVP では同期実行とし、exit 0 で即座に圧縮を許可する。

### 3.6 フック障害時のフォールバック（NFR-10）

全 hook に共通のフォールバック方針:

- hook スクリプトが timeout またはエラー終了した場合、Claude Code はデフォルト動作に戻る
- PreToolUse の場合: Claude Code の `permissions` 設定（allow/ask/deny）に従う → 実質的に `ask` に劣化
- 全操作停止にはならない（graceful degradation）

## 4. Wave 0: 収束条件・停止基準

### 4.1 Green State 定義

| # | 条件 | MVP | 完全実装 | 判定方法 |
|---|------|-----|---------|---------|
| G1 | テスト全パス | ○ | ○ | `pytest` / `npm test` の exit code |
| G2 | lint 全パス | ○ | ○ | `ruff` / `eslint` の exit code（設定がある場合） |
| G3 | 対応可能Issue全解決 | - | ○ | quality-auditor の出力パース（下記詳細） |
| G4 | 仕様差分ゼロ | - | ○ | 仕様ドリフト検知ツール |
| G5 | セキュリティチェック通過 | ○ | ○ | `npm audit` / `pip audit` + シークレットスキャン + 危険パターン検出 |

**G3 詳細（v1.2 更新）**: 旧定義「Critical Issue 0件」を拡張。

| 重要度 | Green State 条件 |
|--------|-----------------|
| Critical | 0件（必須） |
| Warning | PG/SE級は修正済み。PM級は「理由付き保留（deferred）」としてログ記録 |
| Info | PG級は修正済み。SE級以上は「理由付き保留（deferred）」としてログ記録 |

「放置」は禁止。残存 Issue は全て `deferred` + 理由が記録されている状態を Green State とする。これにより Warning/Info の暗黙的放置を防止し、PG/SE級の軽微な問題はループ内で自動解決される。

## 6. Wave 2: ループ統合

### 6.1 `/full-review` コマンド改修

現行の Phase 1-5 構造を維持しつつ、自動ループを追加:

```
Phase 0 (新規): ループ初期化
  - .claude/lam-loop-state.json を生成
  - 対象ファイル、max_iterations を設定

Phase 1: 並列監査（変更なし）
Phase 2: レポート統合 + PG/SE/PM 分類（追加）
Phase 3: 修正
  - PG級: 自動修正（承認不要）
  - SE級: 修正 + ログ記録
  - PM級: 停止 + エスカレーション

Phase 4: 検証（Green State 判定に変更）
  - テスト実行 + lint 実行
  - Green State 達成 → Phase 5 へ
  - 未達 → Claude が「まだ終わっていない」旨を応答 → Stop hook が block → Phase 1 に戻る

Phase 5: 完了報告 + ループログ出力
  - 状態ファイル削除
  - ループログを .claude/logs/ に保存
```

### 6.2 MVP と完全実装の差分

| 機能 | MVP | 完全実装 |
|------|-----|---------|
| ループ起動 | `/full-review` 実行時に手動初期化 | Stop hook で自動判定 |
| 差分チェック | 変更ファイルのみ | 変更ファイル + 依存先 |
| 収束判定 | テスト + lint | Green State 4条件 |

## 7. Wave 3: ドキュメント自動追従

### 7.1 `/ship` コマンド改修

Phase 2（Doc Sync チェック）を強化:

```
Phase 2 (強化): Doc Sync チェック
  1. 変更ファイルを PG/SE/PM に分類
  2. PG級のみ → Doc Sync スキップ
  3. SE/PM級の変更あり:
     a. 対応する docs/specs/ を特定
     b. doc-writer エージェントで更新案を生成（差分形式）
     c. 更新案をユーザーに提示
     d. 承認 → 更新を適用
     e. PM級の設計判断を検出 → ADR 起票を提案
```

### 7.2 PostToolUse hook との連携

PostToolUse hook は `src/` 配下の Edit/Write を検出し、`.claude/doc-sync-flag` にフラグを立てる。`/ship` 実行時にこのフラグを参照する。

MVP では `/ship` 時のみトリガー。完全実装では PostToolUse から doc-writer を即時起動。

**重複防止**: `.claude/doc-sync-flag` にはファイルパスを記録する。同一セッション内で既に提案済みのファイルは再提案しない（AC-4.6: 承認疲れ防止）。

### 7.3 仕様ドリフト検知（P4-FR-4）

既存の仕様と実装の乖離を検出する仕組み。Doc Sync（変更後の更新案生成）とは異なり、**既に存在する乖離**を発見する。

- **MVP**: `/full-review` Phase 1 の並列監査に「仕様ドリフトチェック」を含める。quality-auditor が `docs/specs/` と実装の整合性を確認する（現行の Phase 1 の延長）
- **完全実装**: 定期スキャン（`/loop` との連携）

## 8. Wave 4: TDD 内省

### 8.1 パターン記録の実装

PostToolUse hook がテスト失敗→成功のパターンを検出し、ログに記録する。なお、Claude Code はツール失敗時に `PostToolUseFailure` イベント（`error`, `is_interrupt` フィールド付き）を別途発火するため、完全実装ではこれも活用しパターン検出精度を向上させる:

- 記録先: `docs/memos/tdd-patterns/`（MVP: テキストファイル）
- ファイル名: `YYYY-MM-DD-pattern-NNN.md`

パターン記録のフォーマット:
```markdown
# TDD Pattern: [パターン名]

**観測日**: YYYY-MM-DD
**観測回数**: 1
**ファイル**: src/xxx.py
**失敗内容**: [テスト失敗メッセージ]
**修正内容**: [何を変更したか]
**根本原因**: [なぜ最初に失敗したか]
```

### 8.2 信頼度モデルの実装

MVP では手動照合。完全実装での自動照合設計:

```
1. パターン記録時に既存パターンとの類似度をチェック
   - ファイルパス・エラーメッセージのキーワード一致
2. 類似パターンが見つかれば観測回数をインクリメント
3. 閾値到達時のアクション（初期値: 3回）:
   - ルール候補を .claude/rules/auto-generated/draft-NNN.md に生成
   - PM級としてユーザーに承認要求
```

### 8.3 ディレクトリ構造

```
.claude/rules/
├── core-identity.md          # 人間作成（既存）
├── phase-rules.md            # 人間作成（既存、Wave 1 で更新）
├── security-commands.md      # 人間作成（既存、Wave 1 で更新）
├── decision-making.md        # 人間作成（既存）
├── permission-levels.md      # 人間作成（Wave 1 で新規）
└── auto-generated/           # 自動生成（Wave 4 で新規）
    └── (承認済みルールが配置される)
```

### 8.4 ルール寿命管理（P3-FR-5）

MVP ではスコープ外。完全実装での方針:

- 各自動生成ルールに `last_matched` 日付をメタデータとして記録
- `/daily` 実行時に 90 日以上未使用のルールを棚卸し対象として通知（※v4.3.0 で削除済み）
- ルール削除は PM 級（人間承認必須）

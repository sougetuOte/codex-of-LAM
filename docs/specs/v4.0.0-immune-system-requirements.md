## 1. エグゼクティブサマリー

### 1.1 目的

LAM（Living Architect Model）を **第2世代（憲法型ハーネス）** から **第3世代（免疫系）** へ進化させる。第4世代（マルチエージェント群・自己組織化）には飛ばず、第2世代の堅実な土台の上に自己修復・学習能力を積む。

### 1.2 コアメタファー: 免疫系

| 免疫系の機能 | LAM v4.0.0 での対応 |
|---|---|
| 自己/非自己の判別 | 権限等級（PG/SE/PM）による変更分類 |
| 常時パトロール | ループ統合（差分チェック＋収束判定） |
| 抗体生成・獲得免疫 | TDD内省→ルール自動生成 |
| 恒常性維持 | ドキュメント自動追従 |
| 炎症反応・停止 | 収束条件・停止基準 |

### 1.3 5つの柱（実装優先順）

| Wave | 柱 | 新規性 | 実装難度 |
|------|---|--------|---------|
| 0（基盤設計） | P5: 収束条件・停止基準 | 中 | 中 |
| 1 | P1: 権限等級システム（PG/SE/PM） | 中 | 中 |
| 2 | P2: ループ統合 | 中 | 高 |
| 3 | P4: ドキュメント自動追従 | 中 | 中 |
| 4 | P3: TDD内省→ルール自動生成 | 高（既存実装なし） | 高 |

> **ブリーフからの意図的な逸脱について**: 発注書の優先順位は「P1→P2→P3→P4→P5」だが、本要件定義では「P5(基盤)→P1→P2→P4→P3」に変更した。理由は以下の2点:
>
> 1. **P5（収束条件）の前提条件性**: 両リサーチレポートが合意した DP-5「収束条件設計が全柱の前提条件」に基づき、P5 の設計を Wave 0 に繰り上げた。P5 なしに P2（ループ統合）は機能しない。ただし P5 の実装は各 Wave と同時進行であり、Wave 0 は設計のみ。
> 2. **P3 と P4 の入れ替え**: P3（TDD内省）は「既存実装なし」の最も実験的な柱であり、P1・P2 の運用経験を十分に積んだ上で着手すべきと判断した。P4（ドキュメント自動追従）は `/ship` の Doc Sync チェック強化として既存機能の延長線上にあり、P3 より先に着手する方が段階的導入の原則（DP-10）に合致する。

### 1.4 スコープ外

- 第4世代要素の全面導入（Agent Teams常用、常時マルチエージェント通信）
- 外部ワークフローエンジン（LangGraph等）の導入
- 公開ベンチマーク対応の製品化
- src/ 配下の実装コード（LAMはプロトコルセットであり、アプリケーションコードを含まない）

## 2. 設計原則

### 2.1 両リサーチレポートが合意した5原則

以下は Claude Deep Research と ChatGPT Deep Research の調査結果が独立して到達した合意点であり、v4.0.0 の設計原則として正式に採用する。

| # | 原則 | 根拠 |
|---|------|------|
| DP-1 | **統合こそがイノベーション** | 5柱を統合した既存フレームワークは存在しない。LAMの価値は一貫した統合にある |
| DP-2 | **第3世代を堅実に固める** | 第4世代に飛ばない。第2世代の土台の上にコスト効率を維持しながら自己修復能力を積む |
| DP-3 | **TDD内省→ルール化が最高価値フロンティア** | このクローズドループを実装する既存システムは存在しない |
| DP-4 | **コスト意識をファーストクラスに** | $100/月で実現可能だが、モデルルーティングの工夫が必須 |
| DP-5 | **収束条件設計が全柱の前提条件** | 停止基準なしには他の4柱は機能しない |

### 2.2 追加設計原則（4つの論点判断から導出）

AoT + Three Agents Model による分析の結果、以下の原則を追加採用する。

| # | 原則 | 導出元 |
|---|------|--------|
| DP-6 | **ネイティブ機能優先、フックで補完、外部はエスケープハッチ** | 論点1（ループの複雑さ）、論点3（分類器実装） |
| DP-7 | **三層防衛: 憲法プロンプト → ネイティブ権限 → フック** | 論点2（ルール強制方法） |
| DP-8 | **運用KPI主軸、公開ベンチは参考値** | 論点4（評価方法） |
| DP-9 | **迷ったらSE級（安全側に倒す）** | requirement-analyst 分析より |
| DP-10 | **各柱にMVP/完全実装の二段構え** | 段階的導入・独立ロールバック |

### 2.3 アーキテクチャパターン

全柱に共通する実装アーキテクチャ:

```
基盤層:  Claude Code ネイティブ権限モデル (allow/ask/deny)
補完層:  Hooks (PreToolUse/PostToolUse/Stop 等)
制御層:  LAM 固有ロジック (.claude/rules/, commands/, agents/)
外殻層:  憲法的プロンプティング (CLAUDE.md)
```

## 3. アーキテクチャ決定（ADR要約）

### 3.1 論点1: ループ設計戦略

**決定**: マスターループ優先。LangGraph等の外部エンジンは導入しない。

- Claude Code のシングルスレッド・マスターループを活用
- Stop hook で Ralph Wiggum パターンを実装（公式プラグイン `plugins/ralph-wiggum/` の利用 or 独自実装は design フェーズで判断）
- SESSION_STATE.md への外部化で疑似チェックポイントを実現
- コンテキスト残量 20% で強制セーブポイント

**却下した選択肢**: LangGraph導入（デバッグ複雑性増加、Claude Code固有機能との統合コストが正当化できない）

### 3.2 論点2: ルール強制方法

**決定**: 三層防衛を採用。

| 層 | 方式 | 役割 |
|---|---|---|
| Layer 0 | 憲法的プロンプティング | 意図の解釈・曖昧な判断のガイド |
| Layer 1 | Claude Code ネイティブ権限 | ハードな境界設定（deny/ask/allow） |
| Layer 2 | PreToolUse フック | 動的・文脈依存の判断 |

**却下した選択肢**: フック単独への集中（フック障害時のフォールバックがない）

### 3.3 論点3: PG/SE/PM分類器の実装方式

**決定**: ネイティブ権限モデルを粗粒度境界に、PreToolUseフックを細粒度判定に使う二段構成。

- settings.json でPG/SE/PMごとの基本 allow/deny を静的設定
- PreToolUse フックでタスクコンテキストを解析し動的に上書き

**却下した選択肢**: 外部エンジン即時導入（現時点ではオーバーエンジニアリング）

### 3.4 論点4: 評価方法

**決定**: 運用KPIを主軸（Tier 1）とし、公開ベンチは参考値（Tier 2）。

- Tier 1: タスク完了率、ループ平均イテレーション数、フック介入率、コンテキスト枯渇頻度
- Tier 2: SWE-bench等は「汚染リスクあり・参考値」と明記
- KPI定義は docs/specs/ でバージョン管理

## 5. 柱別要件定義

### 5.0 柱5: 収束条件・停止基準（全柱の基盤）

#### User Story

「開発者として、自動ループが際限なく回り続けてコストとコンテキストを浪費することなく、適切なタイミングで停止または人間にエスカレーションしてほしい。なぜなら、$100/月の予算内で安全に自律性を高めたいから。」

#### Problem Statement

- **現状**: ループの停止判断は人間が手動で行う。`/full-review` は最大2回の再試行でハードコードされている
- **理想**: 品質指標・コスト指標・信頼度指標を組み合わせた収束条件で自動判定。人間は「収束しない」時だけ呼ばれる

#### 機能要件

| ID | 要件 | MVP | 完全実装 |
|----|------|-----|---------|
| P5-FR-1 | Green State の定義: テスト全パス + lint全パス + Critical/Warning Issue 0件（deferred 済み含まず） + 仕様差分ゼロ。Info は阻害しない。詳細は `green-state-definition.md` G3、判断基準は `code-quality-guideline.md` | テスト + lint のみ | 4条件すべて |
| P5-FR-2 | 最大イテレーション上限（デフォルト5回、設定可能） | 固定5回 | 設定可能 |
| P5-FR-3 | コンテキスト残量チェック（20%以下で強制セーブ＋停止） | Stop hook で実装 | 自動 |
| P5-FR-4 | エスカレーション条件: 同一Issue 2サイクル連続再発、PM級変更検出、テスト数減少 | テスト数減少検出 | 全条件 |
| P5-FR-5 | ループログの構造化出力（投入差分、検出問題、修正内容、収束理由） | テキスト出力 | JSON構造化 |

#### 受入条件

- [ ] AC-5.1: Green State 4条件のうち少なくとも2条件（テスト＋lint）が自動判定される
- [ ] AC-5.2: ループが6回以上回らない（デフォルト上限5回）
- [ ] AC-5.3: コンテキスト残量20%以下で停止または `/quick-save` 提案がなされる
- [ ] AC-5.4: テスト数が前サイクルより減少した場合にエスカレーションされる
- [ ] AC-5.5: ループ完了時にログ（何回回ったか、何を修正したか）が出力される

#### Claude Code 機能との対応

| 要件 | Claude Code 機能 |
|------|-----------------|
| P5-FR-2 (イテレーション上限) | Stop hook + `stop_hook_active` による無限ループ防止 |
| P5-FR-3 (コンテキスト残量) | `PreCompact` hook event / 環境変数 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` |
| P5-FR-5 (ログ出力) | PostToolUse hook (command型) でログ書き出し |

#### リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| 収束条件が厳しすぎる（false negative: 完了しているのに止まらない） | コスト浪費、コンテキスト枯渇 | 最大イテレーション上限（P5-FR-2）で強制停止 |
| 収束条件が緩すぎる（false positive: 未完了なのに止まる） | 品質低下 | MVP ではテスト + lint の2条件から開始し、段階的に条件追加 |
| Stop hook 自体の障害 | ループが停止できない | `stop_hook_active` フラグによる二重停止防止 + フック timeout 設定 |
| テスト数減少検出の誤検出（リファクタリングでテスト統合時） | 不要なエスカレーション | エスカレーション時に「テスト統合の可能性」を併記して人間に判断を委ねる |

### 5.1 柱1: 権限等級システム（PG/SE/PM）

#### User Story

「開発者として、typo修正やlint違反のような自明な修正はAIに自動でやらせ、仕様変更のような重大な判断だけ人間が関与したい。なぜなら、全件手動承認は生産性のボトルネックだから。」

#### Problem Statement

- **現状**: 全ての変更（typo修正もAPI設計変更も）が同じ承認フローを通る
- **理想**: 変更のリスクレベルに応じて自動実行・事後報告・事前承認が分岐する

#### 権限等級の定義

| 等級 | 役割 | 人間の関与 | 典型例 |
|------|------|-----------|--------|
| **PG** | 自動修正・報告不要 | なし | フォーマット修正、typo修正、lint違反修正、テスト失敗の自明な修正、import整理 |
| **SE** | 修正後に報告 | 事後レビュー | テストの追加、内部リファクタリング（公開API不変）、仕様書の細部更新、依存パッケージの minor update |
| **PM** | 判断を仰ぐ | 事前承認 | 仕様変更、アーキテクチャ変更、フェーズの巻き戻し、ルールの追加・変更、公開APIの変更、依存パッケージの major update |

**共通原則**: 「迷ったらSE級（上位等級に丸める）」

#### フェーズとの二軸設計

| | PLANNING | BUILDING | AUDITING |
|--|----------|----------|----------|
| PG | - | 自動修正可 | **自動修正可**（v3.x では修正禁止だった） |
| SE | - | 修正後報告 | 修正後報告 |
| PM | 承認ゲート | 承認ゲート | 承認ゲート |

> AUDITING での PG 権限修正許可は v4.0.0 最大の運用変更。

#### 機能要件

| ID | 要件 | MVP | 完全実装 |
|----|------|-----|---------|
| P1-FR-1 | PG/SE/PM の分類基準を `.claude/rules/` に明文化 | 静的テーブル | 動的判定ロジック |
| P1-FR-2 | Claude Code ネイティブ権限で PG の allow リストを設定 | settings.json 手動設定 | フック連動での動的権限変更（※プロファイル自動切替は公式未サポート） |
| P1-FR-3 | PreToolUse フックで SE/PM の動的判定 | prompt 型フック | command 型 + prompt 型ハイブリッド |
| P1-FR-4 | AUDITING フェーズでの PG 修正許可 | phase-rules.md 更新 | フック連動 |
| P1-FR-5 | 等級判定ログの記録（何をどの等級で処理したか） | テキスト出力 | JSON 構造化 |
| P1-FR-6 | サブエージェントごとの権限等級デフォルト設定 | エージェント定義に等級記載 | フロントマターで model/allowed-tools 制御 |

#### 受入条件

- [ ] AC-1.1: PG 等級の変更（lint修正、typo修正）が人間承認なしで実行される
- [ ] AC-1.2: PM 等級の変更（仕様変更）は必ず人間承認を経由する
- [ ] AC-1.3: SE 等級の変更は実行後に変更内容が報告される
- [ ] AC-1.4: 分類基準が `.claude/rules/` に明文化されている
- [ ] AC-1.5: AUDITING フェーズで PG 権限の修正が実行可能である
- [ ] AC-1.6: 判定に迷った場合、SE 等級（上位等級）に丸められる

#### Claude Code 機能との対応

| 要件 | Claude Code 機能 |
|------|-----------------|
| P1-FR-2 | `permissions.allow`: `Bash(npm run lint)`, `Bash(npm run format)` 等 |
| P1-FR-3 | PreToolUse hook (prompt型): タスクコンテキスト解析 |
| P1-FR-4 | PreToolUse hook: フェーズ状態を読み取り権限を動的変更 |
| P1-FR-6 | スキル/エージェント frontmatter: `model`, `allowed-tools`, `hooks` |

#### リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| PG 等級の誤分類（実はPM級の変更をPGと判定） | 意図しない仕様変更 | 「迷ったらSE」原則 + テスト数減少検出 |
| フック障害で全操作ブロック | 作業停止 | フック基本動作を `ask`（確認要求）にし、障害時は確認ダイアログに劣化 |
| 権限設定と仕様のドリフト | セキュリティホール | `/ship` の Doc Sync チェックで検出 |

### 5.2 柱2: ループ統合（full-review + 構造チェック）

#### User Story

「開発者として、コードレビュー→修正→構造チェックのサイクルを手動で繰り返すのではなく、AIが自動で収束するまで回してほしい。なぜなら、1回のレビューでは全問題が出ず、修正が新問題を生む連鎖があるから。」

#### Problem Statement

- **現状**: `/full-review` は人間が繰り返し実行する必要がある。最大2回のループ。個別問題修正と構造チェックが分離している
- **理想**: 自動収束ループ。差分ベース軽量チェック（毎回）+ フルスキャン（最終）の2層で、品質条件を満たすまで自動的に回る

#### ループフロー

> **設計ヒント**: 毎サイクルは変更ファイルのみの軽量チェック、最終サイクルのみフルスキャンとすることでコストを抑える。詳細は design フェーズで決定。

```
入力: 対象ファイル/ディレクトリ

loop (max P5-FR-2 回) {
  1. full-review (並列監査 → 問題抽出)
  2. 問題を PG/SE/PM に分類 (P1)

  3. PG 問題 → 自動修正
  4. SE 問題 → 修正 + ログ記録
  5. PM 問題 → 停止 + 判断要求

  6. 差分ベース構造チェック (変更ファイルの依存先)
  7. テスト実行 + lint 実行

  8. 収束判定 (P5-FR-1):
     if (Green State) → フルスキャン → 完了
     if (エスカレーション条件) → 停止
     else → 次サイクルへ
}

出力: 完了レポート + ループログ
```

#### 機能要件

| ID | 要件 | MVP | 完全実装 |
|----|------|-----|---------|
| P2-FR-1 | `/full-review` の自動ループ化（Stop hook 統合） | 手動継続確認 | Ralph Wiggum パターン自動ループ |
| P2-FR-2 | 差分ベース構造チェック（毎サイクル） | 変更ファイルのみ | 変更ファイル + 依存先 |
| P2-FR-3 | フルスキャン（最終サイクル） | 既存 `/full-review` Phase 4 相当 | 仕様差分チェック含む |
| P2-FR-4 | 問題の権限等級分類（柱1連携） | テキスト表示 | 自動分岐（PG自動修正/PM停止） |
| P2-FR-5 | ループ間の差分追跡（何が変わったか） | git diff ベース | 構造化差分ログ |
| P2-FR-6 | `/loop` コマンドとの連携（定期パトロール） | なし | `/loop 30m /full-review src/` |

#### 受入条件

- [ ] AC-2.1: ループが収束条件を満たしたとき自動停止する
- [ ] AC-2.2: PG 級の問題は自動修正される（人間承認不要）
- [ ] AC-2.3: PM 級の問題が検出されたときループが停止し人間にエスカレーションされる
- [ ] AC-2.4: 毎サイクルで差分ベースの構造チェックが実行される
- [ ] AC-2.5: 最終サイクルでフルスキャンが実行される
- [ ] AC-2.6: ループ完了時に完了レポートが出力される

#### Claude Code 機能との対応

| 要件 | Claude Code 機能 |
|------|-----------------|
| P2-FR-1 | Stop hook + `stop_hook_active` フラグ |
| P2-FR-2 | PostToolUse hook (command型): `git diff` 結果をチェックスクリプトに渡す |
| P2-FR-6 | `/loop` コマンド（v2.1.71+）|

#### リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| 自動修正の連鎖が予期しない方向に進む | コード品質低下、修正が新バグを生む | PG 級のみ自動修正。SE/PM は停止 + 報告 |
| 差分チェックの境界が不明確 | 検出漏れまたは過剰検出 | MVP では「変更ファイルのみ」から開始し、依存先追跡は段階的に拡張 |
| ループの暴走（修正→新問題→修正の無限連鎖） | コスト浪費、コンテキスト枯渇 | P5-FR-2 の最大イテレーション上限 + 同一 Issue 再発検出（P5-FR-4） |
| フルスキャンのコスト | 大規模プロジェクトでの時間・トークン消費 | フルスキャンは最終サイクルのみ。差分チェックで代替可能な範囲を段階的に拡大 |

### 5.3 柱3: TDD内省→ルール自動生成

#### User Story

「開発者として、TDDサイクルで繰り返し犯す同じ種類の失敗が自動的に学習され、次回から防止されてほしい。なぜなら、同じ失敗を何度も繰り返すのは非効率だから。」

#### Problem Statement

- **現状**: `.claude/rules/` は人間が手動で管理。プロジェクト固有のアンチパターンは暗黙知のまま
- **理想**: TDD失敗パターンから自動的にルール候補が生成され、信頼度に応じて段階的に正式化される

#### 信頼度モデル

繰り返し観測されたパターンほど信頼度が高い。観測回数に応じて段階的にアクションを引き上げる（初期値: 1回=記録、2回以上=ルール候補生成）。閾値は運用データに基づき調整する（v1 の3回から v2 で2回に引き下げ。`/retro` が人間実行であり誤爆リスクが低いため）。

> **重要な設計判断**: MVPでは全ルール追加をPM級（人間承認必須）とする。ルールの自動追加はPG級に見えて実はPM級（プロジェクト全体の振る舞いを変える）であるため。

#### 内省パイプライン

```
TDD実装 → テスト失敗 → 修正 → テスト成功
              |
              v
      PostToolUse hook: 失敗→成功のパターンを検出
              |
              v
      内省: 「なぜ最初に失敗したか」を分析
              |
              v
      パターン抽出 + 既存パターンとの照合
              |
              v
      信頼度に基づく処理:
        1回目 → 記録
        2回目 → 通知
        2回目 → ルール候補生成 + 承認要求
```

#### 機能要件

| ID | 要件 | MVP | 完全実装 |
|----|------|-----|---------|
| P3-FR-1 | TDD失敗パターンの記録 | `docs/artifacts/tdd-patterns/` にテキスト | JSON構造化コーパス |
| P3-FR-2 | パターンの照合と信頼度カウント | 手動照合 | 自動照合 |
| P3-FR-3 | ルール候補の生成（2回以上のパターン） | テキスト提案 | `.claude/rules/auto-generated/` へのファイル生成 |
| P3-FR-4 | ルール候補の承認フロー（PM級） | ユーザー手動追記 | 承認→自動配置 |
| P3-FR-5 | ルールの寿命管理（長期未使用ルールの棚卸し） | なし | 定期棚卸し提案 |
| P3-FR-6 | 人間作成ルールと自動生成ルールの分離 | ディレクトリ分離 | メタデータ付き管理 |

#### 受入条件

- [ ] AC-3.1: TDDで同一パターンの失敗が2回以上発生したとき、ルール候補が生成される
- [ ] AC-3.2: ルール候補は人間の承認なしに `.claude/rules/` に追加されない（MVP）
- [ ] AC-3.3: 失敗パターンが `docs/artifacts/tdd-patterns/` に記録される
- [ ] AC-3.4: 自動生成ルールと人間作成ルールが明確に分離されている
- [ ] AC-3.5: 信頼度（観測回数）がパターンごとに追跡される

#### Claude Code 機能との対応

| 要件 | Claude Code 機能 |
|------|-----------------|
| P3-FR-1 | PostToolUse hook (command型): テスト結果のパターン記録 |
| P3-FR-6 | `.claude/rules/auto-generated/` ディレクトリ（自動ロード対象） |

#### リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| 過学習（偶然の失敗を恒久ルールに） | 不要なルール肥大化 | 信頼度モデル（2回閾値）+ PM級承認必須 |
| 誤った一般化 | 間違ったルール適用 | ルール候補に「根拠」（失敗ログへの参照）を付与 |
| ルール肥大化（fat rules） | コンテキスト圧迫 | 寿命管理 + 定期棚卸し |

### 5.4 柱4: ドキュメント自動追従

#### User Story

「開発者として、コードを変更したときに関連する仕様書やADRが自動的に更新提案されてほしい。なぜなら、ドキュメントの手動管理は忘れがちで、仕様と実装の乖離（ドリフト）が蓄積するから。」

#### Problem Statement

- **現状**: `/ship` の Doc Sync チェックで「更新が必要か」を提示するが、実際の更新は手動
- **理想**: SE/PM 級の変更時に仕様書・ADR の更新案が自動生成され、人間の承認でマージされる

#### トリガー設計（等級別）

| 変更の等級 | ドキュメント更新 | 理由 |
|-----------|---------------|------|
| PG | トリガーしない | typo/lint修正はドキュメントに影響しない |
| SE | 更新案を自動生成 + 報告 | 内部リファクタリングが仕様に影響する可能性 |
| PM | 更新案を自動生成 + 承認要求 | 仕様変更は必ずドキュメント同期が必要 |

#### 機能要件

| ID | 要件 | MVP | 完全実装 |
|----|------|-----|---------|
| P4-FR-1 | `/ship` Doc Sync チェックの強化（更新案自動生成） | 差分テキスト提示 | パッチ形式で提案 |
| P4-FR-2 | SE/PM 級変更時の仕様書更新トリガー | `/ship` 時のみ | PostToolUse hook で即時 |
| P4-FR-3 | ADR 自動起票（PM級の設計判断検出時） | テンプレート提示 | 下書き自動生成 |
| P4-FR-4 | 仕様ドリフト検知（実装と仕様書の乖離アラート） | `/full-review` 内で手動 | 定期スキャン |
| P4-FR-5 | doc-writer サブエージェントの自律化 | 既存 + 等級連携 | PostToolUse からの自動起動 |

#### 受入条件

- [ ] AC-4.1: PM 級の変更後、関連 `docs/specs/` の更新案が自動生成される
- [ ] AC-4.2: PG 級の変更ではドキュメント更新トリガーが発火しない
- [ ] AC-4.3: ドキュメント更新案は人間の承認なしにマージされない
- [ ] AC-4.4: `/ship` 実行時に Doc Sync チェックが更新案を含んで提示される
- [ ] AC-4.5: 仕様ドリフト（実装と仕様の不一致）が検知可能である
- [ ] AC-4.6: 同一ファイルへの更新提案が1セッション1回を超えない（承認疲れ防止）

#### Claude Code 機能との対応

| 要件 | Claude Code 機能 |
|------|-----------------|
| P4-FR-2 | PostToolUse hook: Edit/Write ツール完了後にドキュメント更新判定 |
| P4-FR-5 | Agent frontmatter: doc-writer の model/allowed-tools 設定 |

#### リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| 承認疲れ（更新提案が頻繁すぎる） | ユーザーが提案を無視するようになる | 同一ファイルへの更新提案は1セッション1回まで（頻度上限） |
| 更新案の品質不足 | 不正確なドキュメント更新 | 更新案は差分形式で提示し、人間がレビュー・マージ |
| トリガーの過剰発火（SE級の判定が広すぎる） | 不要な更新提案の乱発 | MVP では `/ship` 実行時のみトリガー（PostToolUse 即時トリガーは完全実装で段階的導入） |

## 9. Perspective Check (Three Agents 最終レビュー)

### [Affirmative]

- 5柱の統合は既存フレームワークに存在しない真のイノベーション
- Claude Code の最新機能（18フック、4ハンドラ型、権限モデル）を最大限活用しており、自前実装を最小化
- Wave 方式の段階的導入により、各段階で独立した価値を提供し、ロールバック可能
- $100/月の予算制約を全設計判断に組み込んでおり、現実的

### [Critical]

- 柱3（TDD内省→ルール自動生成）は「既存実装なし」の実験的領域。MVPの信頼度モデルが実運用で機能するか未検証
- PreToolUse フックの prompt 型は Haiku で動作するが、分類精度が不十分な場合のフォールバックが「迷ったらSE」のみ。誤分類の定量的なリスク評価がない
- 「ドキュメント自動追従」の SE級トリガーが頻繁に発火すると「承認疲れ」を起こす可能性。トリガー閾値の調整メカニズムが未定義
- auto mode（3月12日予定とされていた）がCHANGELOGに未登場。公式機能への依存計画に不確実性がある

### [Mediator]

- Critical の指摘はすべて妥当。以下の対策を要件に反映済み:
  1. 柱3はWave 4（最後）に配置済み。Wave 1-3の運用経験を踏まえてMVPスコープを再評価する
  2. **反映済み**: フック分類精度の定量評価を NFR-14a として追加（Wave 1 後にベースライン計測）
  3. **反映済み**: ドキュメント更新トリガーの頻度上限を AC-4.6 として追加（1セッション1回まで）、柱4リスクテーブルにも記載
  4. auto mode への依存は計画に含めない。LAM独自のPG/SE/PM実装で進め、auto mode がリリースされた場合は統合を検討する

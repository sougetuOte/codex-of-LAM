## 1. 概要

本文書は LAM v4.0.0 の運用品質を定量的に評価するための KPI（Key Performance Indicator）を定義する。

**方針**: まず計測基盤を整え、ベースラインを確立する。数値目標は運用データに基づいて設定する（DP-8: 運用KPI主軸）。

**ベースライン確立時期**: Wave 2 完了後（ループ統合が稼働し、計測データが蓄積されてから）

## 2. Tier 1: 運用KPI（主軸）

| # | KPI | 定義 | 測定方法 | データソース | 集計単位 |
|---|-----|------|---------|-------------|---------|
| K1 | タスク完了率 | 自動ループで収束したタスク数 / 全タスク数 | ループログの `convergence_reason` が `green_state` であるログ数を集計 | `.claude/logs/loop-*.txt` | 週次 |
| K2 | 平均ループイテレーション数 | ループ完了までの平均サイクル数 | ループログの `total_iterations` の平均値 | `.claude/logs/loop-*.txt` | 週次 |
| K3 | フック介入率 | フックが block/ask した回数 / 全ツール呼び出し回数 | permission.log の block/ask エントリ数 / 全エントリ数 | `.claude/logs/permission.log` | 週次 |
| K4 | コンテキスト枯渇率 | 20%以下で停止したセッション数 / 全セッション数 | ループログの `convergence_reason` が `context_exhaustion` であるログ数 | `.claude/logs/loop-*.txt` | 月次 |
| K5 | 同一Issue再発率 | 2サイクル連続で同一Issue検出数 / 全Issue数 | ループログの iterations 配列内で同一 issue が連続出現する回数 | `.claude/logs/loop-*.txt` | 月次 |

### 2.1 KPI 詳細定義

#### K1: タスク完了率

- **計算式**: `count(convergence_reason == "green_state") / count(all_loops) * 100`
- **理想値**: 高いほど良い（自律的に品質基準を満たせている）
- **異常値**: 50% 未満は収束条件またはフック判定の見直しが必要
- **除外条件**: ユーザーが手動停止したループ（`convergence_reason == "user_abort"`）は分母から除外

#### K2: 平均ループイテレーション数

- **計算式**: `sum(total_iterations) / count(all_completed_loops)`
- **理想値**: 低いほど良い（少ないサイクルで収束=効率的）
- **コスト効率の代理指標**: イテレーション数はトークン消費量と相関する
- **異常値**: 平均 4 以上は修正品質またはチェック精度の見直しが必要

#### K3: フック介入率

- **計算式**: `count(decision in ["block", "ask"]) / count(all_permission_entries) * 100`
- **理想値**: 低すぎず高すぎない（5-15% が目安、ベースライン確立後に調整）
- **高すぎる場合**: PG/SE の分類基準が厳しすぎる可能性
- **低すぎる場合**: PM 級の変更が検出されていない可能性
- **内訳集計**: PG/SE/PM 別の介入数も記録し、等級分布を可視化

#### K4: コンテキスト枯渇率

- **計算式**: `count(convergence_reason == "context_exhaustion") / count(all_sessions) * 100`
- **理想値**: 0% に近いほど良い
- **異常値**: 10% 以上はタスク粒度の見直しまたはコンテキスト管理の改善が必要

#### K5: 同一Issue再発率

- **計算式**: 2サイクル連続で同一キーワードの issue が出現した回数 / 全 issue 検出数 * 100
- **理想値**: 0% に近いほど良い（修正が効果的であることを示す）
- **異常値**: 20% 以上は修正戦略の見直しが必要
- **同一Issue判定**: issue の classification + ファイルパス + エラーメッセージキーワードの一致で判定

## 5. 計測基盤

### 5.1 データソース一覧

| ソース | 形式 | 生成タイミング | Wave |
|--------|------|---------------|------|
| `.claude/logs/loop-*.txt` | テキスト（MVP）/ JSON（完全実装） | `/full-review` ループ完了時 | Wave 2 |
| `.claude/logs/permission.log` | テキスト（MVP）/ JSON（完全実装） | PreToolUse hook 判定時 | Wave 1 |
| `.claude/tdd-patterns.log` | テキスト | PostToolUse hook 検出時 | Wave 2 |
| `SESSION_STATE.md` | Markdown | `/quick-save` 実行時 | 既存 |

### 5.2 集計方法

`/quick-save` の Daily 記録ステップで以下を集計:

1. `.claude/logs/loop-*.txt` を走査し、K1〜K5 を計算
2. `.claude/logs/permission.log` を走査し、K3 の内訳（PG/SE/PM 別）を集計
3. 結果をデイリーアップデートのテンプレートに出力

### 5.3 ベースライン確立手順

1. **Wave 2 完了後**: ループ統合が稼働し、計測データの蓄積を開始
2. **2週間の計測期間**: 日常運用でのデータを収集
3. **ベースライン設定**: 各 KPI の平均値・標準偏差を算出
4. **目標値設定**: ベースラインに基づき、改善目標を設定（例: K2 の平均を 20% 削減）

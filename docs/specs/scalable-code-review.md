# Scalable Code Review: 中規模コードベース対応の full-review 設計

**日付**: 2026-03-14
**ステータス**: 調査・提案（未着手）
**目標規模**: 30万行（10万行以上で効率低下は許容）
**前提**: Claude Opus 4.6 1M コンテキスト、実行時間・トークンコストは気にしない

---

## 1. 背景と問題

### 1.1 現状の full-review の限界

現在の `/full-review` は 4 エージェント並列監査 → 修正 → 再スキャンの収束ループで動作する。
v4.4.1 の `.claude/` 配下（約3,000行）では 4 イテレーションで Green State に到達したが、
以下の「もぐらたたき問題」が顕在化した:

| iter | Before | 原因 |
|------|--------|------|
| 1 | 20 | 初回スキャン |
| 2 | 3 | PM級修正 |
| 3 | **13** | 修正の副作用 + 前回見逃し |
| 4 | 0 | Green State |

**根本原因**: 各エージェントが独立にファイルを読むため、修正の副作用を見逃す。
コード修正エージェントとレビューエージェントが同じコンテキストを共有していない。

### 1.2 規模別の課題

| 規模 | 行数 | 推定トークン | 1M コンテキストに対して | 課題 |
|------|------|-------------|----------------------|------|
| 小 | ~5K | ~15K | 余裕（1.5%） | なし（現行で十分） |
| 中小 | ~30K | ~90K | 余裕（9%） | エージェント間の情報共有 |
| 中 | ~100K | ~300K | 30% 占有 | Lost in the Middle リスク |
| 中大 | ~300K | ~900K | 90% 占有 | コンテキスト溢れ、コンパクション必須 |

### 1.3 1M コンテキストで変わること

- 10万行程度なら全コードをメインコンテキストに載せられる
- 30万行は 1M でもギリギリ〜溢れる → 分割戦略が必須
- コンパクション発動時の情報喪失対策が新たな課題

## 2. 業界調査結果

### 2.1 主要 AI エディター/ツールの手法

| ツール | 大規模対応手法 | コンテキスト管理 |
|--------|-------------|---------------|
| **Codex (OpenAI)** | repo-scale reasoning、クラウドサンドボックスで実行検証 | AGENTS.md + MCP でコンテキスト注入 |
| **Cursor** | tree-sitter AST → embedding → Turbopuffer (VectorDB)、Merkle tree 差分同期 | RAG で関連チャンクのみ取得 |
| **Aider** | tree-sitter Repo Map + NetworkX PageRank ランキング | `--map-tokens` で動的サイズ調整 |
| **GitHub Copilot** | full project context gathering + CodeQL/ESLint 統合 | SARIF 形式で静的解析結果を統一 |
| **Anthropic Code Review** | 5 並列エージェント（コンプライアンス/バグ/履歴/コメント/コード品質） | 各エージェントが独立に必要なコードを取得 |
| **Greptile** | セマンティックグラフ構築、マルチホップ推論 | v3 でエージェンティック RAG |

### 2.2 学術的知見

| 知見 | 出典 | 要点 |
|------|------|------|
| **Lost in the Middle** | Liu et al., TACL 2024 | LLM は入力の先頭と末尾を最もよく利用。中間で精度 30%+ 低下 |
| **AST ベースチャンキング** | cAST, EMNLP 2025 | 固定サイズ分割に対し 3.55倍の改善 |
| **静的解析 + LLM** | arXiv:2506.10330 | 静的解析で問題箇所を特定 → LLM に渡すハイブリッドが高精度 |
| **Map-Reduce for LLM** | ACL 2025 | チャンク間依存を Structured Information Protocol で解決 |
| **Observation Masking** | JetBrains, NeurIPS 2025 | 50%+ コスト削減 + 2.6% 解決率向上を同時達成 |
| **Multi-Agent SE Survey** | ACM TOSEM | エージェント間の sync/out-of-sync 検出が課題 |

### 2.3 Python による事前分析の可能性

以下のツールで LLM に渡す前にコードを分析し、コンテキスト消費を 70-90% 削減可能:

| ツール | 用途 | 出力 |
|--------|------|------|
| `tree-sitter` (py-tree-sitter) | AST 構造抽出、チャンキング | 関数/クラスの境界、シグネチャ |
| `ast` (Python 標準) | Python コードの構造解析 | import、定義、依存関係 |
| `pydeps` / `pyan` | モジュール/関数間の依存グラフ | NetworkX グラフ |
| `ruff` | 高速 lint | JSON / SARIF |
| `semgrep` | パターンベースのセキュリティ検査 | JSON / SARIF |
| `bandit` | Python セキュリティ脆弱性 | JSON |
| `vulture` | デッドコード検出 | テキスト |
| `Repomix` | コードベースの AI フレンドリーパック | 単一テキスト |

## 3. 5つのプラン

### Plan A: Static Pre-filter + Targeted LLM Review

**コンセプト**: Python スクリプトが機械的チェックを担当し、LLM は意味的レビューに集中する。

```
[Phase 0] Python 事前分析
    ├── ruff / semgrep / bandit で静的解析 → JSON レポート
    ├── tree-sitter で AST 構造マップ生成（関数シグネチャ、依存関係）
    ├── 依存グラフ構築（影響範囲の算出）
    └── 変更差分の影響ファイル特定

[Phase 1] LLM レビュー（スコープ限定）
    ├── 静的解析の指摘箇所 + 周辺コードのみをコンテキストに投入
    ├── AST 構造マップで全体像を把握
    └── 意味的レビュー（ロジックバグ、設計問題、仕様ドリフト）

[Phase 2] 統合 + 修正
    └── 静的解析の自動修正 + LLM の意味的修正
```

| 項目 | 評価 |
|------|------|
| **対応規模** | 30万行対応可能。静的解析は行数に依存しない |
| **もぐらたたき** | 静的解析は決定的（同じ入力→同じ出力）で再発しない。LLM 部分のみ再スキャン |
| **コンパクション** | LLM に渡すコンテキストが小さいためリスク低 |
| **精度** | 静的解析で検出可能な問題は高精度。意味的問題は LLM の精度に依存 |
| **実装規模** | 中（Python スクリプト群 + full-review 改修） |
| **メリット** | トークン消費 70-90% 削減、静的解析は偽陰性が少ない、再現性が高い |
| **デメリット** | 静的解析ツールのセットアップが言語依存、ツール未対応の言語では効果半減 |

---

### Plan B: AST-Chunked Map-Reduce

**コンセプト**: コードを AST 境界でチャンクに分割し、各チャンクを並列レビュー → 結果を統合。

```
[Phase 0] チャンキング
    ├── tree-sitter で AST 解析
    ├── 関数/クラス/モジュール単位でチャンク分割
    ├── 各チャンクに「のりしろ」を付与（import 文、クラス定義、型情報）
    └── チャンク間の依存関係グラフを生成

[Phase 1] Map（並列レビュー）
    ├── 各チャンクを独立した LLM エージェントに投入
    ├── エージェントは「のりしろ」で周辺コンテキストを把握
    └── チャンク内の Issue リストを生成

[Phase 2] Reduce（統合）
    ├── 全 Issue リストを統合・重複排除
    ├── チャンク間の整合性チェック（依存グラフを参照）
    └── 統合レポート生成

[Phase 3] 修正 + 再検証
    └── 修正対象チャンクのみ再レビュー
```

| 項目 | 評価 |
|------|------|
| **対応規模** | 30万行対応可能。チャンク数が増えるだけ |
| **もぐらたたき** | Reduce フェーズで横断的チェックを行うことで軽減。ただし完全ではない |
| **コンパクション** | 各チャンクが小さいためリスク低。Reduce フェーズで大量の Issue が集まる場合は注意 |
| **精度** | チャンク内は高精度。チャンク間の問題は「のりしろ」の設計に依存 |
| **実装規模** | 中〜大（チャンキングエンジン + Map-Reduce オーケストレーション） |
| **メリット** | 自然に並列化できる、AST ベースで構文的整合性を保証 |
| **デメリット** | 「のりしろ」の設計が難しい、チャンク間の問題を見逃すリスク |

---

### Plan C: Hierarchical Review（階層的レビュー）

**コンセプト**: ファイル → モジュール → システム の階層で段階的にレビュー。下位の要約を上位に渡す。

```
[Layer 1] ファイルレビュー
    ├── 各ファイルを個別に LLM レビュー
    ├── ファイル内の Issue + ファイルの「概要カード」（責務、公開API、依存先）を生成
    └── 概要カードは 100-200 tokens に圧縮

[Layer 2] モジュールレビュー
    ├── 関連ファイルの「概要カード」群 + モジュール定義を LLM に投入
    ├── モジュール間の API 整合性、依存関係の妥当性をチェック
    └── モジュールの「要約カード」を生成

[Layer 3] システムレビュー
    ├── 全モジュールの「要約カード」+ アーキテクチャ仕様を LLM に投入
    ├── アーキテクチャ整合性、循環依存、仕様ドリフトをチェック
    └── 統合レポート生成

[Layer 4] 修正（逆方向）
    ├── Layer 3 の Issue → 影響モジュール特定 → 影響ファイル特定
    └── 修正対象ファイルのみ再レビュー
```

| 項目 | 評価 |
|------|------|
| **対応規模** | 理論上無制限。各レイヤーのコンテキストは制御可能 |
| **もぐらたたき** | 各レイヤーが異なる粒度の問題を検出。修正の副作用は下位レイヤーで検出 |
| **コンパクション** | 概要カードにより情報が圧縮されているためリスク低 |
| **精度** | 概要カードの品質に大きく依存。圧縮時の情報喪失リスク |
| **実装規模** | 大（3レイヤーの設計、概要カード生成、逆方向修正パス） |
| **メリット** | 自然なコード構造に沿った分析、各レイヤーで適切な粒度の問題を検出 |
| **デメリット** | 実装が複雑、概要カードの品質管理が難しい、レイヤー間の情報喪失 |

---

### Plan D: Dependency-Graph Guided Review（依存グラフ駆動）

**コンセプト**: 依存グラフを構築し、トポロジカル順にレビュー。下流への影響を追跡しながら進む。

```
[Phase 0] グラフ構築
    ├── tree-sitter + pydeps/pyan で依存グラフ構築
    ├── トポロジカルソートで処理順序を決定
    ├── 強連結成分（循環依存）を検出・マーク
    └── PageRank で重要度ランキング

[Phase 1] 順序付きレビュー
    ├── リーフノード（依存先なし）から順にレビュー
    ├── 各ノードのレビュー結果を「契約カード」（公開API、前提条件、保証）として記録
    ├── 上流ノードのレビュー時に、下流の「契約カード」を参照
    └── 契約違反を検出

[Phase 2] 変更影響分析
    ├── 修正対象ノードから上流を辿り、影響範囲を特定
    └── 影響範囲内のノードのみ再レビュー

[Phase 3] 循環依存の特別処理
    └── 強連結成分はまとめて1チャンクとしてレビュー
```

| 項目 | 評価 |
|------|------|
| **対応規模** | 30万行対応可能。グラフのノード数に比例 |
| **もぐらたたき** | 依存方向に沿ったレビューで副作用の伝播を追跡。最も効果的 |
| **コンパクション** | 契約カードが小さいためリスク低。ただしグラフ全体は保持が必要 |
| **精度** | 依存関係に沿った問題検出が非常に高精度。独立した品質問題は見逃しうる |
| **実装規模** | 中〜大（グラフ構築 + トポロジカルソート + 契約カード設計） |
| **メリット** | 修正の副作用追跡に最適、変更影響分析が自然にできる、再レビュー範囲を最小化 |
| **デメリット** | 依存関係のないファイル間の問題を見逃す、循環依存の処理が複雑、動的言語ではグラフ精度が低下 |

---

### Plan E: Hybrid Pipeline（ハイブリッドパイプライン）

**コンセプト**: Plan A〜D の長所を組み合わせた統合パイプライン。

```
[Stage 1] 静的事前分析（Plan A）
    ├── ruff / semgrep / bandit で機械的チェック
    ├── tree-sitter で AST 構造マップ + 依存グラフ生成
    └── 結果: 静的 Issue リスト + 構造メタデータ

[Stage 2] チャンク分割 + 並列レビュー（Plan B + D）
    ├── 依存グラフベースでモジュール単位にチャンキング（Plan D のグラフを利用）
    ├── 各チャンクに「のりしろ」として依存先の契約カードを付与
    ├── 静的解析の指摘箇所を各チャンクに注入（LLM が重点的にチェック）
    └── チャンクごとに並列 LLM レビュー（Map フェーズ）

[Stage 3] 階層的統合（Plan C）
    ├── チャンクレビュー結果をモジュール単位で集約
    ├── モジュール間の整合性チェック
    └── システム全体の仕様ドリフト・アーキテクチャチェック

[Stage 4] 修正 + 影響範囲再レビュー
    ├── PG/SE 級の自動修正
    ├── 依存グラフで影響範囲を特定（Plan D）
    └── 影響チャンクのみ再レビュー（全体再スキャン不要）

[Stage 5] Green State 判定
    ├── 静的解析の再実行（決定的、即座に完了）
    ├── 影響チャンクのレビュー結果確認
    └── 新規 Issue = 0 で Green State
```

| 項目 | 評価 |
|------|------|
| **対応規模** | 30万行対応可能。各ステージが独立にスケール |
| **もぐらたたき** | 静的解析の決定性 + 依存グラフの影響追跡で大幅に軽減 |
| **コンパクション** | 各チャンクが小さく、メタデータは Python 側で管理。LLM コンテキストは最小 |
| **精度** | 静的解析 + AST チャンキング + 依存関係追跡の三重保証 |
| **実装規模** | 大（全ステージの設計と統合） |
| **メリット** | 全アプローチの長所を統合、再レビュー範囲を最小化、スケーラビリティ最高 |
| **デメリット** | 実装が最も複雑、段階的導入が難しい、オーケストレーション自体のバグリスク |

## 5. コンパクション対策

### 5.1 問題

Claude Code の auto-compact はコンテキストの約 95% で発動し、
会話を要約して古いメッセージを置換する。30万行規模のレビューでは:

- 修正中のファイル内容が要約で失われる
- Issue リストが圧縮されて個別 Issue が消える
- 依存関係の文脈が失われて再レビュー時に誤判定が発生

### 5.2 対策

| 対策 | 手法 | 適用プラン |
|------|------|-----------|
| **外部永続化** | Issue リスト、構造メタデータを JSON/MD ファイルに書き出し | 全プラン |
| **チャンク分割** | LLM に渡す単位を小さくし、1チャンクの処理中にコンパクション不要に | B, D, E |
| **Python 側で状態管理** | 依存グラフ、契約カード、影響範囲をPython 変数/ファイルで管理 | A, D, E |
| **セッション分割** | 大きなレビューを複数セッションに分割し、SESSION_STATE.md で引き継ぎ | 全プラン |
| **Observation Masking** | 古い/重要度の低い情報を明示的に削除（JetBrains 方式） | C, E |
| **サマリー先行生成** | 修正前にファイルの「概要カード」を生成し、コンパクション後も参照可能に | C, E |

### 5.3 設計原則

> **「LLM コンテキストに載せるのは判断に必要な情報だけ。事実の保管は外部ファイルに。」**

- Issue リスト → `docs/artifacts/audit-reports/`（既存）
- 構造メタデータ → `.claude/review-state/`（新規）
- 依存グラフ → `.claude/review-state/dependency-graph.json`（新規）
- 契約カード → `.claude/review-state/contracts/`（新規）

## 6. 10万行の壁と段階的効率低下

### 6.1 推定効率カーブ

```
行数        手法              推定効率
~10K        現行 full-review  100%（問題なし）
10K-30K     現行 + Plan A     90%（静的解析で大半カバー）
30K-100K    Plan A + B        80%（チャンキングで対応）
100K-200K   Plan A + B + D    65%（依存グラフ必須、精度低下開始）
200K-300K   Plan E            50%（全ステージ必要、コンパクション頻発）
300K+       Plan E + 分割     40%以下（セッション分割、人間介入増加）
```

### 6.2 効率低下の主因

1. **チャンク間の情報喪失**: 「のりしろ」では補えない暗黙的な依存関係
2. **コンパクションの累積**: 長時間セッションでの情報喪失が蓄積
3. **統合レポートの肥大化**: Issue 数の増加で Reduce フェーズが重くなる
4. **動的言語の限界**: Python / JavaScript の動的ディスパッチは静的解析で追跡困難

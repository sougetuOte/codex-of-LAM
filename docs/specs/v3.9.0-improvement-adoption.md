## 2. V1: `/ship` コマンド

### 目的

日常的なコミットワークフローを構造化し、論理グループ分けコミット・秘密情報チェックを自動化する。

### フェーズ構成（簡素化版: 5フェーズ）

1. **棚卸し**: `git status` + `git diff --stat` で変更ファイルを一覧化。秘密情報（.env, credentials, settings.local.json 等）を検出した場合は警告して除外
2. **Doc Sync チェック**: 実装変更に対応する CHANGELOG / README / 仕様書の更新漏れを確認し、不足があれば追記対象をリスト化
3. **グループ分け + コミット計画**: 変更を論理グループ（例: feat, fix, docs, refactor）に分類し、グループごとのコミットメッセージ案を提示
4. **確認**: コミット計画全体をユーザーに提示し、承認を得る
5. **実行**: 承認後、グループ単位で `git add` + `git commit` を実行

### 制約

- push はしない（push は明示的な別操作）
- dry-run モード: 引数に `dry-run` を指定した場合、Phase 4 まで実行して終了
- 秘密情報パターン: `.env`, `credentials`, `secret`, `token`, `password`, `settings.local.json`

## 3. V2: `/full-review` コマンド

### 目的

並列エージェント監査 + 全修正 + 検証の一気通貫ワークフロー。
L-3 (audit-fix-policy) を Phase 3 に統合。

### フェーズ構成

1. **並列監査**: 独立した観点の reviewer サブエージェントを並列起動
   - 対象ファイル/ディレクトリはユーザー指定
   - エージェント構成はプロジェクト規模に応じて調整可能（デフォルト: `code-reviewer` x2 + `quality-auditor` x1）
   - 各エージェントは独立した監査レポートを生成
2. **レポート統合**: 複数エージェントの結果を統合し、重複排除・重要度分類（Critical / Warning / Info）
3. **全修正（audit-fix-policy）**:
   - 全重篤度の Issue に対応する（Critical のみではない）
   - 対応不可の Issue は理由 + 追跡先 + 暫定対策を明記
   - 仕様ズレが発見された場合は仕様書も同時修正
4. **検証**: テスト + lint 実行で修正の健全性を確認
5. **完了報告**: Before/After の定量比較（検出数、修正数、残存数）

### /auditing との使い分け

- `/auditing`: フェーズ切替コマンド。AUDITING モードに入り、手動で監査対象を指定して段階的に進める
- `/full-review`: ワンショット実行。並列監査→修正→検証を一気に完了する

### 追記先: `.claude/rules/phase-rules.md` BUILDING セクション

```
### TDD 品質チェック
```

- R-4（テスト網羅）: TDD の本質的チェック
- R-2,3,5,6 はプロジェクト固有のため、テンプレートには含めず追加例として示す

## 5. V4: 仕様同期ルール追記

### 追記先: `.claude/rules/phase-rules.md` BUILDING セクション

以下を「### 仕様同期ルール」として追加:

```
### 仕様同期ルール

- S-1: Green 直後に対応する `docs/specs/` を確認し、実装と仕様の乖離がないか検証
- S-3: 仕様書の未実装項目には Phase/Wave マークを付与（暗黙スキップ禁止）
- S-4: Refactor で公開 API/インターフェースが変わった場合、仕様書を即時更新
```

### 選定理由

- S-1: TDD の Green 直後が最も仕様突合しやすいタイミング
- S-3: 「やらない」を明示することで暗黙のスキップを防止
- S-4: Refactor で公開インターフェースが変わることは言語非依存で発生する
- S-2（Protocol外メソッドの明示）は Python 固有のため除外
